## Context

dbdoctor-tools 当前依赖用户手动创建 `.env` 文件来配置环境变量（DBDOCTOR_URL, DBDOCTOR_USER, DBDOCTOR_PASSWORD）。这要求用户：
1. 知道 `.env` 文件的存在
2. 手动复制 `.env.example` 到 `.env`
3. 编辑文件填入正确的配置值

这增加了使用门槛，特别是对于不熟悉环境变量配置的用户。

## Goals / Non-Goals

**Goals:**
- 实现零配置上手：首次使用任意脚本时自动检测配置缺失
- 交互式引导：提示用户输入必要的配置信息
- 自动生成 `.env` 文件：保存配置供后续使用
- 保持向后兼容：已存在的 `.env` 文件和系统环境变量仍然有效

**Non-Goals:**
- 不修改现有脚本的使用方式（所有脚本调用保持不变）
- 不引入外部配置服务或数据库存储
- 不加密 `.env` 文件中的密码（保持现状）

## Decisions

### 1. 在 Config 类初始化时检测并触发自动配置
- **选择**: 在 `Config.__init__()` 中检测环境变量缺失，若缺失则调用初始化向导
- **理由**: 所有脚本都通过 `from common.config import config` 使用配置，这是统一的入口点
- **替代方案**: 在每个脚本开头单独检测（ rejected：代码重复，维护困难）

### 2. 使用标准输入进行交互式配置
- **选择**: 使用 Python 内置 `input()` 函数读取用户输入
- **理由**: 无需额外依赖，跨平台兼容（Windows/Linux/Mac）
- **输入验证**: 对 URL 格式进行基本验证，确保必填项不为空

### 3. 配置加载优先级保持不变
- **系统环境变量 > `.env` 文件 > 交互式输入**
- 若系统环境变量已设置，跳过 `.env` 检测和交互式配置
- 若 `.env` 文件存在且完整，直接加载，不触发交互

### 4. 初始化流程设计
```
Config() 被实例化
    ↓
检测必需环境变量是否存在
    ↓
存在？ → 正常加载，结束
    ↓
不存在
    ↓
检测 .env 文件是否存在
    ↓
存在？ → 尝试加载，加载成功则结束
    ↓
.env 不存在或加载后仍缺失变量
    ↓
触发交互式初始化
    ↓
提示用户输入 DBDOCTOR_URL
提示用户输入 DBDOCTOR_USER
提示用户输入 DBDOCTOR_PASSWORD（输入时隐藏）
    ↓
生成 .env 文件
    ↓
重新加载配置，继续执行
```

## Risks / Trade-offs

| Risk | Mitigation |
|------|------------|
| 非交互式环境（如 CI/CD）中 `input()` 会挂起 | 检测 `sys.stdin.isatty()`，非 TTY 环境抛出明确错误提示 |
| 用户输入错误的配置 | 提供配置验证选项，尝试连接验证（可选） |
| 并发执行时多个进程同时触发初始化 | 文件锁或简单文件存在性检查（后写入者覆盖，可接受） |
| 密码在终端输入时可见 | 使用 `getpass` 模块隐藏密码输入 |

## Migration Plan

无需迁移，此变更为纯新增功能：
1. 已有 `.env` 文件的用户不受影响
2. 已设置系统环境变量的用户不受影响
3. 新用户首次使用时会自动触发配置引导
